当前 demo 以“注册登录闭环”为业务样例，验证 Android、iOS、HarmonyOS 三端“共享业务逻辑、保留原生 UI”的跨平台架构项目。

1. 技术路线：KMP + native UI
2. 整体项目架构：MVVM + 单向数据流
3. 项目模块结构：common、androidApp、iosApp、harmony-kmp-bridge、harmonyApp
4. 各端接入共享业务层的方式
5. 数据持久化
6. 测试文件

## 1. 技术路线：KMP + 原生 UI

这是项目最重要的技术路线。

| 平台 | UI 技术 | 共享业务接入方式 |
| --- | --- | --- |
| Android | Kotlin + Jetpack Compose | 直接调用 KMP common |
| iOS | Swift + SwiftUI | Shared.framework + Swift Adapter |
| HarmonyOS | ArkTS + ArkUI | KNOI Bridge + libkn.so |

Android 的 Compose、iOS 的 SwiftUI、HarmonyOS 的 ArkUI 都有成熟生态，保留原生 UI 更利于体验与后期维护。

账号格式校验、密码规则、验证码有效期、注册登录、会话失效、错误码等业务规则在 `common` 中维护。Android 和 iOS 直接使用 KMP 产物；HarmonyOS 由独立 bridge 编译同一份 `commonMain` 纯业务源码。

## 2. 核心架构：MVVM + 单向数据流

项目采用分层组合架构。三端 UI 层采用原生 MVVM：页面负责展示，平台 ViewModel 负责连接 UI、导航和共享业务。共享业务层采用 MVI 风格的单向数据流。用户操作统一抽象为 LoginAction，由 LoginStore 处理，并输出 LoginState 和 LoginEffect。LoginState 表示当前持续状态，例如账号输入、登录态和错误信息；LoginEffect 表示一次性事件，例如登录成功、登出和会话失效。LoginStore 不直接操作页面导航，它只表达业务结果。Android 的 AuthNavGraph、iOS 的 AuthRouter、HarmonyOS 的 AuthNavigator 分别消费这些业务事件，并使用各自平台的原生导航能力完成页面跳转。因此，三端页面实现不同，但登录、注册、验证码、会话和错误处理规则只维护在 common 中，实现业务一致、UI 原生、架构可扩展。

## 3. 项目模块结构

可以展示这个结构：

```
demo/
├── common/                KMP 共享业务模块
├── androidApp/            Android Compose 原生应用
├── iosApp/                iOS SwiftUI 原生应用
├── harmonyApp/            HarmonyOS ArkUI 原生应用
├── harmony-kmp-bridge/    HarmonyOS 的 KNOI Bridge
├── docs/                  架构与接入文档
└── tools/                 构建、生成与验证脚本
```

逐个说明职责：

- **common**：核心业务层，只放与平台无关的逻辑。
- **androidApp**：Compose 页面、导航、Android 本地存储。
- **iosApp**：SwiftUI 页面、Swift ViewModel、KMP Framework 适配。
- **harmonyApp**：ArkTS/ArkUI 页面和导航。
- **harmony-kmp-bridge**：将 KMP 的 Kotlin 逻辑编译为 HarmonyOS 可调用的 libkn.so。
- **docs**：记录架构边界、平台接入方式和开发流程。

可以强调共享层边界：

> common 不依赖 Android SDK、iOS API、ArkUI 或 Compose。它只关心业务模型和业务规则，因此可以被不同平台复用。

## 4. 各端接入共享业务层

common 主要包含：

- **LoginRules**：账号、验证码、密码等输入规则
- **LoginModels**：账号、会话、资料、错误码、Action、State、Effect
- **LoginStore**：状态机和 Action 分发
- **LoginUseCase**：登录、注册等业务编排
- **AuthRepository**：认证数据接口
- **LocalMockAuthRepository**：本地 Mock 实现
- **LoginFacade**：跨语言调用门面
- JSON 序列化与反序列化逻辑

### Android

Android 本身使用 Kotlin，所以可以直接调用共享模块中的 LoginStore、LoginRules、LoginAction 和 LoginEffect。

Compose 页面通过 LoginViewModel 管理 UI 状态，导航由 Compose Navigation 处理。

### iOS

iOS 通过 KMP 导出的 Shared.framework 调用共享业务。为了不让 SwiftUI 页面直接依赖 Kotlin 的复杂类型，我们使用 LoginFacade 提供简单稳定的业务方法，例如请求验证码、校验验证码、提交登录、登出、读取状态。SharedLoginAdapter 再将 Facade 包装成 Swift 风格接口，Swift ViewModel 最终只依赖 Adapter。

### HarmonyOS

HarmonyOS 的 UI 仍然完全使用 ArkUI 原生实现，但不会再复制一份 ArkTS 登录业务逻辑。harmony-kmp-bridge 将 Kotlin 业务编译成 libkn.so，并通过 KNOI 生成 ArkTS 调用接口。ArkTS 的 KnoiLoginAdapter 只负责调用 Bridge、映射状态、处理持久化和页面提示。真正的账号校验、验证码、注册、登录、会话逻辑仍然执行在 KMP common 中。

HarmonyOS 保留 ArkUI 原生页面，但不复制登录、注册、验证码和会话规则；页面层只负责展示、导航与平台能力适配，业务规则仍只有一份。

## 5. 数据持久化与可恢复性

可以这样讲：

为了模拟真实 App 的行为，项目将账号库、当前 Session、验证码状态和用户资料保存为 JSON。`auth_mock.proto` 是唯一字段契约；当前没有使用生成的 protobuf Message 类，`common` 中的 `MockAuthStoreJson` 是统一的认证 Store 编解码入口。

每个平台的存储介质可以不同，但存储模型和业务行为保持一致。由于 HarmonyOS bridge 不直接提供 protobuf JSON runtime，三端统一复用 common 中按 protobuf JSON 命名规则实现的受约束快照编解码；平台层只读写快照字符串，不自行拼装认证 JSON。

| 平台 | 存储实现 |
| --- | --- |
| Android | SharedPreferences + JSON |
| iOS | UserDefaults + JSON |
| HarmonyOS | ArkTS StorePersister + JSON Snapshot |
| 测试 | 内存数据源 |

这里的架构价值是：

> 业务层只依赖 AuthStoreDataSource 抽象，不关心数据最终写到 SharedPreferences、UserDefaults 还是 HarmonyOS Preferences。认证模块当前只使用本地 mock；如未来接入真实服务，应在独立项目或分支重新设计。

## 6. 测试与质量保障

### 第一层：规则测试

对应 `LoginRulesTest`，当前共 7 条共享规则测试。

测试内容：

- 手机号格式、规范化处理
- 邮箱格式校验
- 验证码必须为 4 位数字
- 密码长度和字母数字组合规则
- 两次密码一致
- Loading 状态下禁止重复提交

讲法：

这一层验证最基础的输入规则。例如同一个手机号、密码、验证码输入，在 Android、iOS、HarmonyOS 上都会调用同一套 LoginRules，因此不会出现某端允许、另一端不允许的情况。

### 第二层：业务流程测试

对应 `LoginUseCaseTest`，当前共 28 条共享认证流程测试。

测试内容：

- 注册成功并创建 Session
- 重复注册
- 验证码错误
- 验证码过期
- 重发验证码替换旧验证码
- 正确与错误密码登录
- 修改密码
- 重置密码
- 完善资料
- 登出
- 会话失效
- 删除账号
- 应用重启后恢复 Session

讲法：

第二层验证完整业务流程。重点不只是"成功情况"，还验证了重复注册、错误密码、验证码过期、会话失效等真实异常场景。

例如验证码测试会先生成一个验证码，再推进模拟时间，验证共享层最终返回"验证码已过期"。这保证了验证码过期判断不是某个页面的倒计时，而是业务层的真实规则。

### 第三层：数据异常与持久化测试

测试内容：

- 空数据
- 损坏的本地 Session
- JSON 序列化/反序列化
- 兼容旧字段命名
- 持久化保存失败

讲法：

第三层验证应用在本地数据不完整、数据损坏或保存失败时，能安全返回明确错误，而不是直接崩溃或保留错误登录态。

此外，`BusinessMockDataSourceTest` 验证后续业务必须通过 `AuthRepository.verifyBusinessAccess()` 获取当前 Session；未登录、登出和会话失效后均返回 `AuthRequired`。Android 另有 Instrumentation Test 验证 `SharedPreferences` 中的登录态可由新的数据源实例恢复。
