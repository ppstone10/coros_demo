# SDD 开发流程规范

## 元数据

- Spec ID：`SDD`
- 状态：已采纳
- 适用范围：本仓库后续所有功能、缺陷修复、重构与影响交付行为的配置变更
- 规范入口：`AGENTS.md`
- 可执行门禁：`./tools/check-sdd.sh`

## 目标

把“规格先行、测试表达规格、实现满足测试、结果可追溯”落实为仓库内可重复执行的开发闭环，减少新 AI 会话依赖历史聊天上下文的问题。

## SDD-001：会话预检

开始工作前必须按顺序读取：

1. `LEARNINGS.md`：持久架构决策、项目边界与已验证的踩坑。
2. `spec/TRACE.md`：已有规格、测试和实现的追溯关系。
3. 与本次任务相关的 `spec/*.md`。
4. 必要时读取 `DESIGN.md`、`REQUIREMENT_NOTES.md` 与 `docs/worklog/` 中的历史证据。

如果入口文件与实际代码冲突，以当前可验证的代码和测试为事实，并把冲突记录为人工审查点；不得静默覆盖。

## SDD-002：先定义 Spec

行为变更开始前，必须新增或更新对应 `spec/<feature>.md`。Spec 至少定义：

- 目标与非目标。
- 适用边界和禁止事项。
- 数据/状态/接口约束。
- 可观察行为与异常行为。
- 验收标准和测试要求。
- 带稳定 ID 的规范条目，例如 `AUTH-LOGIN-001`。

纯文案、注释和不影响行为的机械格式调整可以引用既有 Spec，不强制新增 Spec；只要行为、接口、数据或验收口径发生变化，就必须先更新 Spec。

## SDD-003：预留 TRACE 映射

写测试或实现之前，在 `spec/TRACE.md` 为新增/变更的规范 ID 预留行，状态设为 `⏳`。预留行可以暂时把测试或实现写为 `待补`，但必须明确 Spec ID 与目标位置。

TRACE 状态统一为：

- `⏳`：已定义，尚未完成实现或验证。
- `✅`：测试、实现、验证证据均已补齐。
- `⚠️`：部分完成或存在明确风险，必须写清缺口。
- `🚫`：经人工决定不实现，并记录原因。

## SDD-004：测试是可执行 Spec

实现前先编写或更新测试，并至少运行一次，确认新测试能够捕获缺失行为（失败、无法编译，或由评审确认是针对既有行为的回归测试）。

测试应直接对应规范 ID，覆盖本次范围内的正常路径、边界、错误和状态恢复。若某项无法自动化，Spec 和 TRACE 必须写明人工验收方法，不能把“未执行”记为通过。

## SDD-005：最小实现

只实现使 Spec 与测试通过所需的改动，不把未写入 Spec 的额外行为混入同一轮。实现必须遵守 `LEARNINGS.md` 中的架构、安全和跨端边界。

如果实现过程中发现 Spec 不完整，先回到 Spec 和 TRACE 修订，再继续实现。

## SDD-006：完成追溯

实现完成后更新 `spec/TRACE.md`：

- 使用仓库相对路径定位测试和实现。
- 优先引用稳定的类型、函数或测试名称，避免只依赖易漂移的行号。
- 记录实际验证命令或人工验收入口。
- 只有验证通过且不存在未披露缺口时才标记 `✅`。

## SDD-007：更新 Worklog

每轮实际修改结束后，在 `Codex_worklog.md` 追加且仅追加本轮事实。每轮记录先写一级标题：

```markdown
# YYYY-MM-DD HH:mm — 内容概要
```

- 时间使用项目时区（Asia/Shanghai）、24 小时制并精确到分钟，表示实际写入时间。
- 内容概要用一行说明本轮主要处理事项，避免使用“本轮修改”“问题处理”等无法检索的空泛标题。
- 迁移旧记录时如果无法确认原始写入时间，使用 `# YYYY-MM-DD（时间未记录）— 内容概要`，不得伪造具体时间；该格式只用于历史补录。

标题之后固定包含以下四部分：

- `采纳内容`：已经落地或明确决定的内容。
- `人工审查点`：仍需负责人确认的业务、风险或取舍。
- `验证结果`：实际执行的命令/检查及结果。
- `人工修正点`：尚需人工补齐或修正的事项；没有则明确写“暂无明确人工修正点”。

每条记录引用稳定 Spec ID（例如 `[SDD-007]`），不得把建议或未运行的测试写成已验证事实。同一轮的时间和概要只写在一级标题中，不需要在每条 bullet 重复日期。

## SDD-008：提炼 Learnings

只把跨会话仍然有效的架构决策、边界、已验证踩坑和可复用方法写入 `LEARNINGS.md`。过程记录、一次性命令输出和本轮变更清单只写入 Worklog，避免让 Learnings 退化为第二份流水账。

## SDD-009：关闭门禁

一轮开发只有同时满足以下条件才算完成：

1. Spec 已新增或更新，且规范 ID 稳定。
2. TRACE 已从预留状态更新为真实状态。
3. 测试先于实现建立，并记录实际验证结果。
4. 实现与 Spec 一致，相关测试通过。
5. `./tools/check-sdd.sh` 通过。
6. `Codex_worklog.md` 已追加“真实写入时间 + 内容概要 + 四段记录”。
7. 有持久价值的结论已提炼进 `LEARNINGS.md`；若没有，不为凑数而追加。

紧急修复也不得跳过闭环；可以压缩文档篇幅，但必须在本轮结束前补齐 Spec、TRACE、验证和 Worklog。

## 验收标准

- 根目录存在 AI 可自动发现的 `AGENTS.md`，完整表达 SDD 顺序与完成门禁。
- `spec/README.md` 能解释目录职责、生命周期、命名和状态。
- `spec/TEMPLATE.md` 可直接复制创建新功能 Spec，并包含稳定 ID、边界、行为和测试字段。
- `spec/TRACE.md` 同时支持“预留”和“完成”状态，并映射本规范条目。
- `./tools/check-sdd.sh` 能检查核心入口、固定章节、规范 ID 和文件名一致性。
- `./tools/check-sdd.sh` 能检查每轮 Worklog 都有合法时间/概要标题，且四段顺序完整。
- `README.md` 与 `docs/development-workflow.md` 指向同一套权威流程，不再维护相互冲突的流程副本。
