# App 实习生培训公共要求

本文档需要和个人培训方案一起分发给实习生。个人方案只描述具体项目目标、实现范围、验收标准和测试要求；本文档描述所有方案共用的技术栈、安全边界、认证前置、交付物和评分口径。

本培训的核心任务是：使用 KMP 共享业务层 + Native UI 技术栈，复刻 COROS App 的某个脱敏功能切片。实习生需要复刻产品功能、页面交互、状态流转和业务规则，但不得复刻真实服务端接口、真实协议、真实加密逻辑、真实请求 / 响应结构或真实数据。

## 1. 项目来源与参考资料

- 所有培训方案的开发功能均源自 COROS App 的脱敏功能切片，功能细节可参考 COROS App。
- Android 内测版本获取页：https://workspace.coros.com/admin/sub_admin_faq/changelog?name=android-COROS
- iOS 内测版本获取页：https://workspace.coros.com/admin/sub_admin_faq/changelog?name=ios-COROS
- UI 设计稿请向导师索取 Figma 链接。
- 功能体验只能参考 App 行为和脱敏需求说明；不得接入、复刻或记录真实服务端接口、真实协议、真实请求参数、真实响应结构或真实数据。

## 2. 安全边界

- 开发必须在隔离训练仓库完成，不得接触或引用正式项目源码、真实接口、真实协议、真实密钥、真实用户数据、内部包名、类名、接口域名或业务代号。
- 所有业务数据必须使用本地 mock 数据、本地 fixture 或本地 mock repository 生成，不得接入生产环境、测试环境、内测环境或任何真实服务端。
- mock 数据结构必须使用 Protocol Buffers 定义：每个业务项目至少提供对应的 `.proto` 文件，说明本项目 mock 数据、状态数据和错误场景数据的字段结构。KMP 共享业务层可以把 protobuf message 转换为 domain model，但不得用无结构约束的临时 JSON 替代数据结构定义。
- 不保留服务器 API 说明，不编写真实 URL、真实 path、真实请求头、真实请求参数、真实响应结构或加密签名逻辑。
- mock 数据必须能保存状态：新增、编辑、删除、已读、收藏、草稿、任务进度、登录态等操作需要在 App 本地可恢复。状态结构也必须有对应 protobuf message 定义。持久化可使用 protobuf binary、protobuf JSON、SQLite、DataStore、UserDefaults、KeyValue 存储或等价本地方案。
- 训练代码不依赖当前正式仓库。
- 不设计硬件端、外设协议、外设 SDK 或固件；涉及状态、任务、资源等能力时，只做 App 侧状态消费、任务编排和异常处理模拟。

## 3. 技术栈与架构

- 所有项目统一使用 KMP 共享业务层 + Native UI。
- 端侧从 Android、iOS、鸿蒙中三选一实现 Native UI，可在对应模拟器或虚拟设备上完成开发、Demo 和验收，不强制使用真机。
- 页面架构必须选择 MVVM 或 MVI，不使用 MVP。
- KMP 共享业务层承载业务模型、业务规则、protobuf 数据结构、本地 mock 数据访问、数据访问抽象、状态持久化和核心单元测试。
- Native UI 只消费 KMP 输出的页面状态、错误状态和一次性效果，只负责页面展示、交互和平台能力适配。

## 4. 注册登录前置要求

每位实习生都必须先实现注册 / 登录模块，作为后续业务项目的统一前置能力。第 2 周注册登录验收是硬门禁：未通过注册登录验收的实习生，不能进入业务项目第 3 周开发；补齐后再继续业务项目。

注册登录模块只使用本地 mock 账号数据，不接入任何服务器。模块至少包含：

- 注册：支持账号、密码、测试验证码和注册成功后的本地登录态保存。
- 登录：支持账号密码登录、登录中状态、防重复提交、密码错误提示和登录成功后的会话保存。
- 登出：清理本地登录态；再次访问业务 mock 数据源时应进入未登录状态。
- 会话失效：本地 mock 数据源能模拟会话失效，页面能清理会话并引导重新登录。
- 业务数据访问：后续业务 mock 数据源必须读取当前登录态；未登录时返回明确的未登录错误。
- 状态保存：注册账号、登录态、登出状态和会话失效状态必须在本地可验证、可恢复。
- 测试：至少覆盖注册成功、登录成功、密码错误、未登录访问业务数据、登出后访问业务数据和会话失效。

第 2 周验收脚本：

1. 注册新账号，注册成功后自动保存 `AuthSession`。
2. 使用账号密码登录，登录成功后后续业务 mock 数据源能读到当前登录态。
3. 密码错误、验证码错误、账号不存在时展示明确错误态。
4. 登出后清理本地登录态，再访问业务 mock 数据源返回未登录错误。
5. 模拟会话失效，页面能清理会话并引导重新登录。
6. 认证模块单元测试通过，且无真实账号、真实 token、真实接口、真实密钥或真实响应结构落库。

## 5. 统一 3 个月节奏

| 周期 | 阶段 | 目标 | 产出 |
| --- | --- | --- | --- |
| 第 1 周 | 入门、安全规范与账号域 | 熟悉 Kotlin / Swift、Git、Codex 协作、安全边界、账号状态基础 | 小练习、`Codex_worklog.md`、认证状态草图 |
| 第 2 周 | 注册登录前置模块硬门禁 | 完成本地 mock 注册 / 登录 / 登录态管理，并理解 MVVM / MVI 分层 | 可运行认证模块、认证单测、W2 验收记录 |
| 第 3 周 | 业务需求拆解 | 阅读脱敏 PRD，拆用户场景、数据模型、账号依赖和异常态 | `REQUIREMENT_NOTES.md`、问题清单 |
| 第 4 周 | 业务技术方案 | 设计 domain model、usecase、protobuf mock 数据结构、状态持久化、页面结构和认证拦截 | `DESIGN.md`、测试计划 |
| 第 5-6 周 | 可运行版本开发 | 完成主流程可运行版本 | 可运行 App、基础单测 |
| 第 7-8 周 | 核心规则补齐 | 补异常态、状态机、缓存、重试、边界规则 | 核心 usecase、规则测试 |
| 第 9 周 | 工程化增强 | 优化结构、日志、错误提示、mock 场景切换 | debug 工具、体验补齐 |
| 第 10 周 | 回归与 Review | 修复 review 问题，提高测试覆盖 | review 整改记录 |
| 第 11 周 | 验收准备 | 自测、文档、录屏、答辩材料 | `TEST_REPORT.md`、Demo |
| 第 12 周 | 终期答辩 | 产品验收、代码评审、技术答辩 | 留用评分与复盘 |

## 6. 统一交付物

每份单人方案最终必须交付：

- 可运行 App demo：包含 KMP 共享业务层和 Android / iOS / 鸿蒙三选一的 Native UI，可在对应模拟器或虚拟设备上运行。
- 注册 / 登录 / 登录态管理模块，并能被后续业务 mock 数据源复用。
- 本地 mock 数据源和状态持久化实现，能证明新增、编辑、删除、草稿、收藏、已读、任务进度等项目相关状态可保存。
- `.proto` 文件：定义本项目 mock 数据、状态保存数据和错误场景数据结构，并说明 protobuf message 到 KMP domain model 的转换边界。
- `README.md`：运行方式、项目结构、核心功能。
- `REQUIREMENT_NOTES.md`：需求拆解、已确认规则、待确认问题。
- `DESIGN.md`：KMP 共享业务层边界、MVVM / MVI 架构设计、业务模型、状态流转、账号态依赖、protobuf mock 数据结构和状态持久化方案。
- `TEST_REPORT.md`：测试范围、命令、结果、未覆盖风险。
- `Codex_worklog.md`：Codex 使用记录、采纳内容、人工审查点、验证结果和人工修正点。
- 业务项目标准档统一至少 12 条单元测试；挑战项可追加测试，但不作为标准档最低要求。
- 至少覆盖 4 类异常态：未登录、空数据、错误数据、本地 mock 数据读取失败或状态保存失败。
- 第 2 周注册登录验收记录；未通过者必须附补救记录。

## 7. Codex 协作要求

- 可以使用 Codex 辅助需求拆解、方案设计、代码生成、测试补齐和问题排查。
- 必须审查、验证和修正 Codex 产物，不能提交本人无法解释的生成代码。
- 所有采纳内容、人工审查点、验证结果和修正点必须记录到 `Codex_worklog.md`。
- 不得要求 Codex 生成真实服务端接口、真实加密逻辑、真实请求参数、真实响应结构或真实密钥。

## 8. 统一强度校准

所有项目默认按同一"标准档"主考核强度设计。个人方案里的基础、标准、挑战是完成档位，不是题目级别标签。

| 完成档位 | 统一定义 | 评分用途 |
| --- | --- | --- |
| 基础要求 | 注册登录硬门禁通过，业务主流程可运行，关键异常态可展示，核心测试通过 | 判断是否具备独立完成 App 小功能的最低能力 |
| 标准要求 | 完成项目文档中的主流程、4-6 个核心业务规则或页面状态、至少 12 条单元测试、4 类异常态、本地 mock 状态保存和完整交付文档 | 作为所有实习生的默认横向比较口径 |
| 挑战要求 | 在标准档之外补可复用抽象、复杂边界、额外测试、团队可复用工具或跨端共享沉淀 | 只用于高分区分，不作为合格门槛 |

最终留用讨论只比较标准档完成质量；若有人完成挑战要求，必须单独标注挑战项内容，避免把题目天然难度混入基础评分。
